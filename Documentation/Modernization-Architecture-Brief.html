Architecture Review — Stakeholder Brief

Spring Music
Modernization Plan

An architectural analysis and decomposition roadmap for three P1 user stories: View Catalog, Sort Catalog, and View Single Album.

Prepared: March 2026 Scope: US-1, US-2, US-3 Strategy: Strangler Fig Risk Tier: Low → Medium

Executive Summary

The legacy Spring Music application is a Spring Boot 2.4 monolith with an AngularJS 1.2 frontend — both end-of-life. The three Priority-1 user stories expose three concrete modernization failures: no DTO layer, no server-side sort, and no proper error handling. This brief quantifies the problems and proposes a phased remediation plan requiring no big-bang rewrite.

What We Are Modernizing

Three P1 read-path user stories: browsing the full album catalog (US-1), sorting it by field and direction (US-2), and fetching a single album by ID (US-3).

Core Problem

The JPA Album entity is exposed directly over HTTP, sorting is entirely client-side in a dead JavaScript framework, and a missing album returns null rather than HTTP 404.

Recommended Approach

Strangler Fig pattern. Extract the catalog read path into a dedicated service behind an API gateway. Modernize in three phases — no service downtime required.

Approval Sought

Approval to begin Phase 1 work: DTO layer, server-side sort, HTTP 404 contract fix, and Flyway schema management. Two-week sprint estimate.

Circular Concerns

⚠ Entity-as-DTO Loop

AlbumController serialises the JPA Album entity directly into the HTTP response. Any schema change — renaming a column, adding a field — immediately breaks every API consumer. There is no isolation boundary between persistence and the public contract.

⚠ Profile Bootstrap Loop

SpringApplicationContextInitializer activates the redis profile → Spring loads RedisConfig → which creates the RedisTemplate → which RedisAlbumRepository needs to construct — but the repository's @Profile("redis") was what the initializer enabled. Startup ordering is fragile; a missing CF env var silently selects the wrong database.

User Story AnalysisP1 Stories — Current vs. Target State

Each P1 story is examined against its acceptance criteria, with the specific legacy defect identified and the modernized implementation described.

US-1

View Album Catalog

As a catalog user, I want to view all albums so that I can browse the music inventory.

ACAcceptance CriterionLegacy StatusGap
AC-1 Albums displayed in a 4-column responsive grid Partial AngularJS renders grid; no responsive framework upgrade
AC-2 Each card shows title, artist, year, genre, track count Pass Fields exist; year stored as String not Integer
AC-3 Toggle between grid and list view Partial Implemented in AngularJS; no equivalent in modern stack
AC-4 30 pre-seeded albums on first launch Pass Works but uses fragile ContextRefreshedEvent listener, not a DB migration
AC-5 Page loads within acceptable time Risk findAll() — no pagination. Entire table transferred to client on every load

Legacy Code (the wince)

@GetMapping
public List<Album> list() {
    // Raw JPA entity returned directly — schema = public API contract
    // No pagination. Fetches every row on every request.
    return (List<Album>) repository.findAll();
}

Modernized Implementation

@GetMapping
public Page<AlbumResponse> list(@PageableDefault(size = 20) Pageable pageable) {
    // DTO isolates API from persistence model
    // Pageable: ?page=0&size=20  — only fetches what the client needs
    return catalogService.findAll(pageable).map(AlbumMapper::toResponse);
}

Additional changes: Flyway replaces auto-DDL; seed data becomes V2__seed_albums.sql; React + Vite replaces AngularJS.

US-2
US-2

Sort Album Catalog

As a catalog user, I want to sort albums by different fields so that I can find what I'm looking for faster.

ACAcceptance CriterionLegacy StatusGap
AC-1 Sort by title, artist, release year, or genre Client-only AngularJS orderBy filter — zero server-side implementation
AC-2 Toggle ascending / descending Client-only Same as AC-1 — client sorts a full fetch every time
AC-3 Applies immediately without full page reload Pass Works today because data is already in browser; will break once pagination lands
AC-4 Current sort field and direction visually indicated Pass UI indicator exists; not connected to any server state

Root Cause

The GET /albums endpoint returns an unsorted list. The AngularJS controller holds the full dataset in $scope.albums and applies orderBy as a view-layer filter. Once AC-5 of US-1 is fixed (pagination), this approach collapses entirely — the browser only holds one page, so client-sort produces wrong results. Sorting must move to the server before pagination lands.

Modernized Implementation

// New endpoint contract:  GET /albums?sort=title,asc  or  ?sort=releaseYear,desc
@GetMapping
public Page<AlbumResponse> list(@PageableDefault(sort = "title", direction = ASC) Pageable pageable) {
    return catalogService.findAll(pageable).map(AlbumMapper::toResponse);
}

// CatalogQueryService — single interface, backend-agnostic
public interface CatalogQueryService {
    Page<Album> findAll(Pageable pageable);   // JPA, Mongo, Redis all implement this
    Optional<Album> findById(String id);
}

releaseYear type fix: change from String to int in the entity (Flyway V3__alter_release_year_type.sql). Numeric sort on year is broken today — "9" sorts after "2024" lexicographically.

US-3
US-3

View a Single Album

As a catalog user, I want to retrieve a specific album by its ID so that I can see its full details.

ACAcceptance CriterionLegacy StatusGap
AC-1 Valid ID returns full album details Pass Works; exposes raw entity including internal id vs albumId confusion
AC-2 Non-existent ID returns null/empty — no server error Wrong contract Returns HTTP 200 with a null body. Should be HTTP 404

Legacy Code (the wince)

@GetMapping("/{id}")
public Album getById(@PathVariable String id) {
    // Returns HTTP 200 with body: null
    // Every REST client treats 200 as success — the null is silently swallowed
    return repository.findById(id).orElse(null);
}

Modernized Implementation

@GetMapping("/{id}")
public AlbumResponse getById(@PathVariable String id) {
    return catalogService.findById(id)
        .map(AlbumMapper::toResponse)
        .orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND,
            "Album not found: " + id));
}

// AlbumResponse — decouples API shape from JPA entity
public record AlbumResponse(
    String  id,
    String  title,
    String  artist,
    int     releaseYear,   // was String in legacy
    String  genre,
    int     trackCount
) {}

Part B — The MapDecomposition Plan & Extraction Risk

We recommend the Strangler Fig pattern. The legacy monolith continues serving write operations while the catalog read path is extracted into a new service. Traffic is routed by an API Gateway — no service downtime, no flag-day cutover.

RoadmapThree-Phase Migration Plan

Each phase is independently releasable. Phases do not require the next phase to be approved before delivering value.

Phase 1 — Low Risk

Harden the Monolith

  • Introduce AlbumResponse DTO + AlbumMapper
  • Fix GET /albums/{id} to return HTTP 404
  • Add ?sort + ?page params to GET /albums
  • Fix releaseYear from String to int
  • Replace auto-DDL with Flyway V1 baseline
  • Move seed data to Flyway V2 migration
  • Add @SpringBootTest slice tests for US-1/2/3
Phase 2 — Medium Effort

Extract Catalog Query Service

  • New Spring Boot 3.x read-only microservice
  • API Gateway routes GET /albums* to new service
  • React + Vite frontend replaces AngularJS catalog view
  • OpenAPI / Swagger docs published
  • Shadow-mode testing (run both, compare responses)
  • Legacy GET /albums* marked deprecated
Phase 3 — Higher Risk

Strangle the Monolith

  • Extract Album Write Service (US-4/5/6)
  • Decouple from Cloud Foundry platform binding
  • Replace shared DB with event-driven sync
  • Decommission legacy monolith
  • Replace remaining AngularJS pages

Phase 1 Delivery Checklist

  • 1
    AlbumResponse record + AlbumMapper — zero behaviour change, pure compile-time refactor. Verifiable by existing manual smoke tests.
  • 2
    HTTP 404 on missing album — one-line change to AlbumController.getById(). Fixes the AC-2 contract for US-3.
  • 3
    Server-side sort via Spring Data Pageable — add ?sort query param. US-2 AC-1/2 move from browser to server. Enables future pagination.
  • 4
    releaseYear type migration — Flyway V3 ALTER COLUMN. Fixes silent string-sort bug on year field.
  • 5
    Flyway V1 baseline + V2 seed — replace generate-ddl: true and AlbumRepositoryPopulator. Schema is now version-controlled.
  • 6
    Integration tests — @SpringBootTest slices covering all three user stories against H2. These are the regression gate for Phases 2 and 3.

Approval RequestedKey Decisions for Stakeholders

The following decisions are needed before Phase 1 work begins.

Decision 1 — DTO Contract

Confirm that the AlbumResponse shape (removing redundant albumId field, changing releaseYear to integer) is acceptable to all existing API consumers.

Impact if deferred: the entity-as-DTO problem compounds with every subsequent schema change.

Decision 2 — Frontend Technology

Confirm React + Vite as the target frontend stack to replace AngularJS 1.2.16. Alternatively: Vue 3, Svelte, or Angular 17.

Impact if deferred: AngularJS reaches end-of-support for its last LTS security patches. The frontend cannot consume paginated/sorted API responses until it is replaced.

Decision 3 — Flyway Baseline

Approve replacing spring.jpa.generate-ddl: true with Flyway. This requires a one-time baseline migration to be applied to all existing non-H2 environments.

Impact if deferred: schema drift between environments cannot be reliably controlled.

Decision 4 — Phase 2 Scope

Confirm that Phase 2 (extracting Catalog Query Service + API Gateway) is in scope for this modernization programme, or whether Phase 1 hardening alone meets the current delivery objective.

Note: Phase 1 delivers immediate value independently of Phase 2 approval.
